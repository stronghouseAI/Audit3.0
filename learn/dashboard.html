📋 Architectural Guide: Audit 3.0

System Blueprint: Production-ready, asynchronous Python processing engine with strict structural mapping and data persistence.

1. System Topography & File Structure

The system operates entirely out of your local decoupled workspace directory, cleanly separating analytical application logic from binary data footprints:

~/Documents/googleDev/Audit3.0
├── qa_auditor.py       # Core AsyncIO batch coordinator & inference processing engine
├── notifier.py         # Outbound Twilio WhatsApp Sandbox messaging client
├── database.py         # Relational SQLite transaction management and tables
├── audit_history.db    # File-based localized SQLite relational database file
└── transcripts/        # Target ingestion directory containing multi-minute binary recordings

2. Asynchronous Flow Control (The Traffic Cop)

To safely maximize processing throughput across bulk assets without triggering network rejection codes, the coordinator applies a strict concurrency boundary:

# Enforced inside qa_auditor.py via:
semaphore = asyncio.Semaphore(2)

# Flow Execution:
# [Bulk .mp3 Queue] ──► [asyncio.Semaphore(2)] ──► Max 2 Workers active simultaneously

Because SQLite is traditionally synchronous, the data pipeline preserves core event-loop velocity by farming database writes off to background worker pools using thread isolation wrappers:

await asyncio.to_thread(save_audit_result, ...)

3. The Relational Data Schema

Data models map directly from incoming strict Pydantic structures down to explicit columns inside audit_history.db across two coupled tables:

Table: agents

Dimension Layer

  • agent_id (TEXT, PK)
  • phone_number (TEXT)
  • created_at (DATETIME)
Table: compliance_audits

Factual Metric Layer

  • audit_id (TEXT, PK)
  • agent_id (FK)
  • audio_filename (TEXT)
  • score (INTEGER, 1-10 Bounds)
  • justification (TEXT)
  • breach_detected (BOOLEAN)
